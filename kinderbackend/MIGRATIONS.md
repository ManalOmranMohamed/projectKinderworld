# Database Migrations

This backend now uses Alembic for schema management.

## Prerequisites

- Python environment with `alembic` installed
- `DATABASE_URL` set for PostgreSQL, or leave it unset to use the local SQLite fallback
- For PostgreSQL, install a driver:

```bash
pip install "psycopg[binary]"
```

Example:

```bash
set DATABASE_URL=postgresql+psycopg://postgres:postgres@localhost:5432/kinderbackend
```

## Fresh Database

Run the latest schema on a new database:

```bash
cd kinderbackend
python -m alembic upgrade head
```

Then start the API normally.

This works for both SQLite and PostgreSQL.

## Existing Legacy Database

If your database already has the current schema from the old `create_all` / `ALTER TABLE` startup flow, back it up first and then mark it as migrated:

```bash
cd kinderbackend
python -m alembic stamp head
```

Use `stamp` only when the database schema already matches the current models.

## Create a New Migration

After changing SQLAlchemy models:

```bash
cd kinderbackend
python -m alembic revision --autogenerate -m "describe_change"
python -m alembic upgrade head
```

Review autogenerated migrations before applying them, especially on SQLite.
PostgreSQL should be treated as the primary production target.

## Check Current Revision

```bash
cd kinderbackend
python -m alembic current
python -m alembic history
```

## Startup Behavior

The app no longer mutates the schema on startup.

- If required tables are missing, startup fails fast with a migration command hint.
- If a legacy database is detected without `alembic_version`, startup logs a warning and continues.

## PostgreSQL Notes

- `postgres://...` URLs are normalized automatically to `postgresql+psycopg://...`
- SQLite-only connection pragmas are applied only when the active database is SQLite
- Alembic batch mode is enabled only for SQLite migrations
